# Plan 07-02: Stop Swarm UI and Visible Error Hardening

<objective>
Add a real user-facing stop action to the workspace cockpit and harden renderer error display so failed real runtimes are visible, safe, and recoverable.
</objective>

<context>
Read before executing:

- `.planning/phases/07-shutdown-hardening-and-demo-readiness/07-CONTEXT.md`
- `.planning/phases/07-shutdown-hardening-and-demo-readiness/07-01-SUMMARY.md` if present
- `renderer/workspace.html`
- `renderer/app.js`
- `renderer/style.css`
- `preload.js`
- `src/swarm-ipc.js`
- `scripts/smoke-workspace-orchestration.js`
- `scripts/smoke-workspace-ui.js`
- `AGENTS.md`
</context>

<scope>
## In Scope

- `STOP SWARM` button in mission bar.
- Renderer stop handler using preload orchestration stop.
- Stop button disabled/active state.
- Safe feed/pane rendering for mission, spawn, Sentinel, and TTS errors.
- Static smoke coverage for stop UI and error hooks.

## Out of Scope

- Main-process lifecycle cleanup internals.
- Demo preflight/runbook.
- Real GUI screenshots unless practical.
</scope>

<threat_model>
Potential issues:

- Stop button remains enabled when nothing is running.
- Stop action resets UI but leaves real processes alive.
- Error messages leak sensitive paths/env/token values.
- Launch overlay remains stuck after stop/error.

Mitigations:

- Derive button state from mission activity and active pane statuses.
- Route stop through the existing preload orchestration stop API.
- Sanitize/truncate renderer-displayed error text.
- Always clear overlay on stop and error paths.
</threat_model>
