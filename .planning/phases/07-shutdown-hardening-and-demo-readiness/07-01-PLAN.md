# Plan 07-01: Idempotent Shutdown and Process Cleanup

<objective>
Harden the process lifecycle so user-stop, runtime-switch, mission-complete, and app-close can all stop active child processes and watchers safely without duplicate kills, hangs, or Electron crashes.
</objective>

<scope>
## In Scope

- Idempotent `stop()` behavior in `src/swarm.js`.
- Idempotent active registry cleanup in `src/swarm-ipc.js`.
- Electron lifecycle cleanup hooks in `main.js`.
- Fake-runtime smoke coverage for repeated stop and cleanup ordering.

## Out of Scope

- Renderer stop button UI.
- Demo runbook/preflight.
- Real Claude/Codex mission execution.
- Mixed runtime execution.
</scope>

<threat_model>
Potential issues:

- Multiple Electron lifecycle events call cleanup more than once.
- A pty process ignores kill and blocks shutdown.
- Sentinel cleanup throws during app quit.
- Mission-complete cleanup accidentally emits duplicate stop/error events.

Mitigations:

- Track stopping/stopped state and make repeated stop calls no-op safe.
- Bound async cleanup with a timeout or best-effort fallback.
- Catch cleanup failures and emit/send warnings instead of throwing during quit.
- Smoke-test repeated stop and app-close paths with injected fake processes/watchers.
</threat_model>

<tasks>
<task id="T1" type="auto">
  <title>Make swarm core stop idempotent and bounded</title>
  <files>
    <file>src/swarm.js</file>
  </files>
  <action>
    Update the swarm core so repeated `stop(reason)` calls are safe. Track whether a mission is active/stopping, avoid duplicate kill attempts after the process map is cleared, and return a structured result such as `{ ok, stopped, reason }`. Catch child kill errors and emit operational `agent:error` events without throwing.
  </action>
  <verify>
    Run `node --check src/swarm.js`.
  </verify>
  <acceptance_criteria>
    - Calling stop twice does not throw.
    - Active child processes are killed at most once per stop sequence.
    - Kill errors emit events but do not crash the caller.
  </acceptance_criteria>
</task>

<task id="T2" type="auto">
  <title>Harden orchestration IPC cleanup</title>
  <files>
    <file>src/swarm-ipc.js</file>
  </files>
  <action>
    Refactor `stopActive(reason)` so it is idempotent, disposes event subscriptions and Sentinel before clearing the active registry, handles concurrent stop calls, and sends cleanup warnings when needed. Preserve mission-complete behavior without emitting an extra user-visible stop event.
  </action>
  <verify>
    Run `node --check src/swarm-ipc.js`.
  </verify>
  <acceptance_criteria>
    - Repeated IPC stop calls are safe.
    - Sentinel stop runs once per active mission.
    - Stop failures are reported as warnings and do not reject app-close cleanup unless truly unrecoverable.
  </acceptance_criteria>
</task>

<task id="T3" type="auto">
  <title>Wire Electron quit lifecycle cleanup</title>
  <files>
    <file>main.js</file>
  </files>
  <action>
    Update Electron lifecycle handling so cleanup is attempted from app close/quit paths, not only `window-all-closed`. Keep quit behavior normal for Windows. Avoid blocking indefinitely; if cleanup is async, use a guarded helper that can be called from multiple lifecycle hooks.
  </action>
  <verify>
    Run `node --check main.js`.
  </verify>
  <acceptance_criteria>
    - App close and quit paths call the centralized cleanup helper.
    - Cleanup can run more than once without duplicate effects.
    - Electron quit is not prevented indefinitely.
  </acceptance_criteria>
</task>
