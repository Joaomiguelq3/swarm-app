# Plan 07-03: Demo Preflight and Runbook

<objective>
Create explicit demo-readiness assets: a real-runtime preflight command and a concise runbook for the Claude Code and Codex demo flows.
</objective>

<context>
Read before executing:

- `.planning/phases/07-shutdown-hardening-and-demo-readiness/07-CONTEXT.md`
- `.planning/phases/07-shutdown-hardening-and-demo-readiness/07-01-SUMMARY.md` if present
- `.planning/phases/07-shutdown-hardening-and-demo-readiness/07-02-SUMMARY.md` if present
- `src/runtimes.js`
- `package.json`
- `README.md` if present
- `AGENTS.md`
</context>

<scope>
## In Scope

- `scripts/demo-preflight.js` that checks local runtime binary availability.
- `npm` script for demo preflight.
- `docs/demo-runbook.md` with two real demo flows.
- Documentation of manual rehearsal evidence expectations.

## Out of Scope

- Running real Claude/Codex missions in automated `check`.
- Creating demo applications inside this repo.
- Packaging/installer docs.
</scope>

<threat_model>
Potential issues:

- Demo preflight leaks environment details.
- Automated checks fail on machines without Claude/Codex even when development checks should pass.
- Runbook overstates fake-runtime coverage as real demo readiness.
- Demo setup pollutes the SWARM repository.

Mitigations:

- Print only command availability/version output and high-level failures.
- Keep preflight script separate from `check`.
- Explicitly separate automated and manual evidence.
- Direct demo workspaces outside this repo by default.
</threat_model>

<tasks>
<task id="T1" type="auto">
  <title>Add demo preflight script</title>
  <files>
    <file>scripts/demo-preflight.js</file>

  </files>
  <action>
    Create a Node script that uses `child_process.spawnSync` to run `claude --version` and `codex --version`, reports pass/fail per runtime, and exits non-zero if either is unavailable. Register it as `demo:preflight`. Do not add it to `check`.

  <verify>
    Run `node --check scripts/demo-preflight.js`.
  </verify>
  <acceptance_criteria>
    - Script checks both runtime commands from `src/runtimes.js` or equivalent canonical config.
    - Script does not print environment variables or secrets.
    - `npm.cmd run check` does not depend on the script passing.
  </acceptance_criteria>
</task>
