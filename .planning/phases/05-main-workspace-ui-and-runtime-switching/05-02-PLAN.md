# Plan 05-02: Runtime Controls and Pane Lifecycle State

<objective>
Make the node panel operational for Phase 5: runtime/model dropdowns reflect the active workspace, model options switch by runtime, changes persist through the existing workspace API, pane runtime/status indicators update, and runtime switching calls a stop-active-processes hook before applying config changes.
</objective>

<scope>
## In Scope

- Runtime dropdown and model dropdown behavior.
- Renderer runtime metadata map matching `src/runtimes.js`.
- Model option refresh when runtime changes.
- Persisting runtime/model changes through preload workspace APIs.
- Named stop-active-processes hook before runtime switch.
- Local pane state reset and runtime indicator refresh.
- Feed entries for workspace loaded, stop hook, runtime changed, model changed, and placeholder launch.
- Visual support for all required pane statuses.

## Out of Scope

- Real process kill/spawn.
- Agent lifecycle event IPC.
- File watcher events.
- Launch overlay/TTS.
</scope>

<threat_model>
Potential issues:

- Runtime/model UI drifts from persistence validation.
- Runtime switch applies before lifecycle hook runs.
- Placeholder launch looks like real work.
- API errors leave UI half-updated.

Mitigations:

- Mirror `src/runtimes.js` values exactly and smoke-test the model strings.
- Use an explicit `await stopActiveProcesses()` call before `updateRuntime`.
- Keep launch disabled or feed copy explicit that orchestration is pending.
- On API errors, keep prior local workspace state and show inline/feed error.
</threat_model>
