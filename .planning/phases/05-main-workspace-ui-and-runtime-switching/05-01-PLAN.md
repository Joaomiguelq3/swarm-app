# Plan 05-01: Workspace Route and Cockpit Shell

<objective>
Connect saved workspace cards to the main workspace route and replace the placeholder workspace page with the full cockpit shell: workspace tabs, three terminal pane surfaces, mission bar, feed, map/files region, and node panel placeholders.
</objective>

<scope>
## In Scope

- Home card click/keyboard navigation to `workspace.html?id=<workspaceId>`.
- Route-aware renderer initialization for Home vs Workspace.
- `workspace.html` DOM replacement with the main cockpit shell.
- Workspace loading by id through preload workspace APIs.
- Inline invalid/missing workspace state.
- Workspace tab strip showing saved workspaces.
- Static pane, mission bar, feed, map/files, and node panel regions.

## Out of Scope

- Runtime/model dropdown persistence.
- Real process stopping.
- Real xterm.js terminal attachment.
- Agent spawning, mission decomposition, file watcher feed, launch overlay, and TTS lifecycle.
</scope>

<threat_model>
Potential issues:

- Home route breaks because `renderer/app.js` starts handling multiple pages.
- Workspace id from the query string is trusted too much.
- User-provided workspace names/paths are rendered unsafely.
- Layout becomes decorative rather than operational.

Mitigations:

- Branch initialization by `document.body.dataset.route` and keep Home initialization isolated.
- Treat the id only as a lookup key against `window.swarm.workspaces.list()` results.
- Use DOM creation and `textContent` for workspace values.
- Keep full-viewport dense panels and stable dimensions; no nested cards.
</threat_model>
