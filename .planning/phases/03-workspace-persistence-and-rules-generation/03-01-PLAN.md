# Plan 03-01: Workspace Persistence Core

<objective>
Implement the UI-independent workspace persistence layer for WRK-01, WRK-02, WRK-03, RUN-03, and PRS-01, including `workspaces.json`, runtime/model validation, `brain.json` bootstrap, and runtime-specific rules file generation.
</objective>

<context>
Read before executing:

- `.planning/PROJECT.md`
- `.planning/REQUIREMENTS.md`
- `.planning/ROADMAP.md`
- `.planning/phases/03-workspace-persistence-and-rules-generation/03-CONTEXT.md`
- `.planning/phases/03-workspace-persistence-and-rules-generation/03-RESEARCH.md`
- `.planning/phases/02-runtime-scout-sentinel-tts-and-terminal-core/02-VERIFICATION.md`
- `src/runtimes.js`
- `AGENTS.md`
- `CLAUDE.md`
</context>

<scope>
## In Scope

- Core workspace persistence module.
- `workspaces.json` load/save with explicit appData override for tests.
- Workspace create/list/update/touch APIs.
- Runtime/model validation.
- `brain.json`, `CLAUDE.md`, and `AGENTS.md` generation without overwrites.
- Smoke tests that use temporary directories.

## Out of Scope

- Full Home UI workspace cards.
- Native folder picker UI.
- Runtime switch process kill/restart behavior.
- Real Claude/Codex process spawning.
</scope>

<threat_model>
Potential issues:

- Path traversal or unexpected write target if the workspace path is not normalized.
- Accidental overwrite of user-authored `brain.json`, `CLAUDE.md`, or `AGENTS.md`.
- Corrupt `workspaces.json` causing silent data loss.

Mitigations:

- Resolve and validate all workspace paths before writing bootstrap files.
- Use "create if missing" semantics and never overwrite existing bootstrap/rules files.
- Fail clearly on invalid JSON rather than replacing it with an empty workspace list.
</threat_model>

<tasks>
<task id="T1" type="auto">
  <title>Create workspace persistence module</title>
  <files>
    <file>src/workspaces.js</file>
  </files>
  <action>
    Create `src/workspaces.js` using CommonJS. Export functions for:
    - resolving the app data directory from `{ app, appDataPath }`, Electron `app.getPath("userData")`, or `%APPDATA%\\swarm` fallback
    - reading and writing `{ workspaces: [] }` from `workspaces.json`
    - listing workspaces sorted by `lastAccess` descending
    - creating a workspace from `{ path, runtime, model, app, appDataPath }`
    - updating runtime and optional model by workspace id
    - updating model by workspace id
    - touching `lastAccess` by workspace id

    Records must contain only `id`, `name`, `path`, `runtime`, `model`, and `lastAccess` as first-class schema fields. Use `crypto.randomUUID()` for ids when available. Validate runtime through `src/runtimes.js`; default model to the selected runtime's first model when omitted.
  </action>
  <verify>
    Run `node --check src/workspaces.js`.
  </verify>
  <acceptance_criteria>
    - Missing workspace storage initializes to `{ "workspaces": [] }`.
    - Invalid runtime is rejected.
    - Missing model receives a runtime-specific default.
    - Workspace records contain all Phase 3 schema fields.
  </acceptance_criteria>
</task>

<task id="T2" type="auto">
  <title>Add workspace bootstrap generation</title>
  <files>
    <file>src/workspaces.js</file>
  </files>
  <action>
    In `src/workspaces.js`, add shared bootstrap logic used by create and runtime update:
    - create workspace root directory only if it is the selected existing path or a test-created path that exists; do not invent parent folders for arbitrary paths
    - create `brain.json` if missing with valid minimal JSON placeholders
    - create the current runtime rules file if missing
    - use existing root `CLAUDE.md` and `AGENTS.md` as templates when available
    - preserve existing `brain.json`, `CLAUDE.md`, and `AGENTS.md`
    - return metadata describing created and preserved files

    Runtime update must create the newly required rules file without deleting the previous runtime's rules file.
  </action>
  <verify>
    Run `node --check src/workspaces.js`.
  </verify>
  <acceptance_criteria>
    - Creating a Codex workspace creates or preserves `AGENTS.md`.
    - Creating a Claude workspace creates or preserves `CLAUDE.md`.
    - Switching runtime creates the other rules file when missing.
    - Existing user files are never overwritten.
  </acceptance_criteria>
</task>
