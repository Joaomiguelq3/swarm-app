# Plan 04-02: Workspace Cards and Create Flow

<objective>
Connect the Home UI to Phase 3 workspace persistence so saved workspaces render as compact runtime-badged cards and users can start the new workspace flow from the Home screen.
</objective>

<context>
Read before executing:

- `.planning/phases/04-home-ui-and-visual-system/04-CONTEXT.md`
- `.planning/phases/04-home-ui-and-visual-system/04-RESEARCH.md`
- `.planning/phases/04-home-ui-and-visual-system/04-01-SUMMARY.md` if present
- `.planning/phases/03-workspace-persistence-and-rules-generation/03-VERIFICATION.md`
- `renderer/home.html`
- `renderer/style.css`
- `renderer/app.js`
- `preload.js`
- `AGENTS.md`
</context>

<scope>
## In Scope

- Workspace loading through `window.swarm.workspaces.list()`.
- Workspace cards with required metadata.
- Runtime badge mapping for Claude Code and Codex.
- New workspace modal/form using path + runtime and workspace create API.
- Loading, empty, and inline error states.

## Out of Scope

- Native folder dialog if it requires main-process expansion beyond the existing API.
- Workspace route navigation into full cockpit.
- Runtime switching UI.
- Terminal panes or agent processes.
</scope>

<threat_model>
Potential issues:

- Renderer might attempt direct file access for path validation.
- User-provided path might be rendered unsafely.
- Failed workspace create could break the entire Home UI.

Mitigations:

- Call only the preload workspace API.
- Use DOM text APIs, not HTML string injection for user-provided values.
- Catch and display errors inline while keeping the Home UI interactive.
</threat_model>
